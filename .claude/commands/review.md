# 代码审查

## 用法
/review
/review <文件路径或功能模块名>

## 执行步骤

### 审查范围
- 无参数：审查当前 git diff（未提交的改动）
- 有参数：审查指定文件或目录

### 审查维度

#### 安全性（最优先）
- Server Actions 是否都调用了 `getSession()` 验证
- 用户输入是否在写入数据库前做了校验
- 是否有 SQL 注入风险（Prisma 已防护，但检查原始查询）
- Cookie 是否设置了 `httpOnly` 和 `secure`
- 密码是否明文存储或输出到日志

#### 正确性
- 数据库查询是否处理了 `null` 返回值
- 异步操作是否都有 `await`
- Server Component 是否误用了 `useState`/`useEffect`
- 分页和过滤逻辑边界情况（空数据、第一页、最后一页）

#### 性能
- 是否有 N+1 查询（循环内查数据库）
- 大列表是否做了分页
- 图片是否用了 `next/image`
- 是否有不必要的 `"use client"` 导致客户端包变大

#### 规范符合度
- 是否符合 CLAUDE.md 的目录约定
- 返回格式是否统一 `{ success, data/error }`
- 是否有 `any` 类型
- 组件命名是否 PascalCase，函数命名是否 camelCase

### 输出格式

🔴 **必须修改**（会导致 bug 或安全漏洞）
- 文件:行号 — 问题描述 + 修复建议

🟡 **建议修改**（代码质量或规范问题）
- 文件:行号 — 问题描述 + 修复建议

🟢 **可选优化**（锦上添花）
- 文件:行号 — 优化思路

⬜ **无问题** — 列出审查过的文件

不要描述代码做了什么，只说为什么有问题。
审查完后询问：是否需要自动修复 🔴 级别的问题？

# Bug 修复流程

## 用法
/fix <bug 描述、报错信息或 GitHub Issue 编号>

## 执行步骤

### 1. 复现问题
- 如果是 Issue 编号，通过 GitHub MCP 读取完整错误信息和复现步骤
- 如果是报错信息，定位到具体文件和行号
- **不要直接修改代码**，先确认问题根因

### 2. 诊断（列出假设）
分析可能的原因，给出 2-3 个假设，例如：
假设 A：Session 验证时 cookie 名称不一致
假设 B：Prisma 查询缺少 include 关联数据
假设 C：客户端组件在 SSR 时访问了 window 对象

### 3. 验证根因
针对每个假设，运行验证命令或添加临时日志确认。
确定根因后告知用户，**等确认后再修改代码**。

### 4. 最小化修复
- 只修改造成问题的代码，不顺手重构无关部分
- 如果修复需要改动超过 3 个文件，先拆分为独立 PR 说明

### 5. 验证修复
```bash
npx tsc --noEmit          # 确认类型无误
npm run build             # 确认构建通过
```

针对该 bug 的具体验证：
- 按原报错的复现步骤重新操作，确认问题消失
- 检查相关的边界情况（空数据、未登录、并发等）

### 6. 回归检查

检查修改是否影响以下关键路径：
- [ ] 首页文章列表正常加载
- [ ] 文章详情页正常显示
- [ ] 管理后台登录正常
- [ ] 新建/编辑文章正常保存
- [ ] 标签筛选正常

### 7. 提交

```bash
git add <涉及的文件>
git commit -m "fix: <问题描述>

根因：<一句话说明为什么会出现这个问题>
修复：<一句话说明怎么修的>"
```

### 8. 完成汇报

输出：
- 根本原因（一句话）
- 修改了哪些文件的哪些部分
- 如何验证修复（操作步骤）
- 是否有潜在的类似问题需要关注
